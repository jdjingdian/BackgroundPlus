## 上下文

当前 BTM 管理界面已包含“登录时打开”和“允许在后台”两个分组，但它们直接消费分类结果，导致以下问题：
- 用户无法看到 dump 的完整集合，排查分类或筛选问题时缺乏参照。
- 列表中可能出现未确认分类或开关关闭条目，影响“可管理项”语义。

本次变更仅调整展示分组与筛选口径，不改动 helper 安装流程、详情页导航路径和开关写入能力。

## 目标 / 非目标

**目标：**
- 在现有两组之外新增“完整 Dump 数据”分组，展示解析后的完整条目集合。
- “登录时打开”和“允许在后台”两组只展示“分类已确认且开关开启”的条目。
- 保持现有列表行交互（开关、详情跳转）与数据刷新路径兼容。

**非目标：**
- 不重构 dump 解析器的底层协议与字段定义。
- 不新增新的写入动作或批量操作。
- 不修改 helper 安装/卸载相关 UI 与流程。

## 决策

1. 采用“单一原始数据源 + 三个派生视图”模型
- 决策：先保留完整 dump 条目数组，再在 ViewModel 层构建三个列表（登录时打开、允许在后台、完整 dump）。
- 原因：避免多处重复解析与分散过滤逻辑，降低分组条件变更成本。
- 备选方案：
  - 在 UI 层临时过滤：实现快，但条件分散在视图中，后续维护成本高。
  - 解析阶段直接丢弃不符合条件条目：会丢失“完整 dump”所需数据，不满足新需求。

2. 对前两组统一应用 `isCategoryConfirmed && isEnabled` 过滤门槛
- 决策：把“分类确认”和“开关开启”作为前两组硬性可见条件。
- 原因：保证这两组的语义是“当前生效且可管理”的条目，减少噪声。
- 备选方案：
  - 仅过滤分类不确定，不过滤开关关闭：仍会混入大量当前未生效条目。
  - 仅在 UI 标灰不隐藏：会保留噪声，不符合用户提出的“只展示”要求。

3. 未确认分类条目集中在“完整 dump”中可见，并提供状态提示
- 决策：当存在未确认分类条目时，不再打散进前两组，只在完整 dump 列表中呈现，并延续“分类不完整”提示。
- 原因：既保留可观测性，又不污染主操作分组。
- 备选方案：
  - 直接忽略未确认条目：可见性不足，不利于排查。

## 风险 / 权衡

- [风险] 过滤规则变严后，前两组可能出现“看起来缺数据”的感知变化。
  -> 缓解：在空状态文案中明确“仅显示已确认且已开启项目”，并引导查看“完整 Dump 数据”。

- [风险] 某些条目开启状态字段缺失或格式异常，导致误过滤。
  -> 缓解：将状态解析失败视为未开启，并在完整 dump 中保留原始条目供核查。

- [权衡] 新增第三组会增加侧栏信息密度。
  -> 缓解：保持排序稳定（登录时打开、允许在后台、完整 Dump 数据）并复用现有分组样式。

## Migration Plan

1. 在 ViewModel 增加完整条目集合与三组派生列表计算逻辑。
2. 更新侧栏分组定义与主列表数据源映射。
3. 更新空状态/提示文案，覆盖新过滤语义。
4. 回归验证：
   - helper 已安装且有混合条目（已启用/未启用、已确认/未确认）
   - 两个主分组空状态
   - 完整 dump 列表条目完整性
5. 回滚策略：若出现严重可用性回归，可回退到仅两分组与原过滤规则实现。
