## 上下文

当前 BTM 管理的读路径已经稳定：App 通过 helper 调用 `sfltool dumpbtm` 获取原始数据并解析展示。  
但写路径尚未落地到系统：行内开关仅修改 ViewModel 覆盖态，删除仅操作内存适配器，导致 UI 操作与系统真实状态不一致。

本次设计目标是在不破坏现有 UI 交互（Dry-Run、风险确认、结果页）的前提下，补齐受控写链路，并将特权写操作统一收敛到 helper。

## 目标 / 非目标

**目标：**

- 建立 App ↔ helper 的写操作协议（路由、请求/响应、错误语义、能力协商）。
- 支持“单点删除”真实执行，并复用现有 Dry-Run 与风险分级。
- 支持“单条开关”真实写入，写后自动回刷 dump。
- 当 helper 无法识别当前系统写 schema 时，强制降级为只读并给出明确提示。

**非目标：**

- 不在本次变更中改造列表 UI 结构、动画和分组逻辑。
- 不在本次变更中支持批量删除或批量开关。
- 不引入绕过 helper 的主程序特权写路径。

## 决策

### 决策 1：继续读写分离，写路径全部走 helper

- 选择：保留 `dumpbtm` 作为读来源；新增 helper 写路由承接开关/删除。
- 原因：与既有特权边界一致，避免主程序直接接触高权限写入。
- 备选：
  - 主程序直接写系统存储：违反当前 helper 边界约束，放弃。
  - 使用 `sfltool` 直接完成单条写：工具本身不提供单点删除/单条开关语义，放弃。

### 决策 2：写能力显式协商，默认只读

- 选择：helper capability 增加 `supportsWriteOperations` 与 `writeSchemaVersion`。
- 原因：BTM 私有结构存在系统版本漂移，必须先协商再允许写入。
- 备选：
  - 固定假设所有系统都可写：风险不可控，放弃。

### 决策 3：删除执行维持“计划在 App，执行在 helper”

- 选择：App 继续生成 DeletePlan 与风险等级；helper 负责按最终计划执行真实写入。
- 原因：App 已有成熟的 Dry-Run 与确认模型，可减少行为变更。
- 备选：
  - 全部下沉 helper：会复制规则并增加协议复杂度，暂不采用。

### 决策 4：写失败必须可恢复

- 选择：写前备份、写后校验、失败回执标准化；UI 失败时回滚本地状态并显示错误。
- 原因：系统级写入风险高，必须具备可观测和可恢复性。

## 风险 / 权衡

- [私有 schema 漂移] → 通过 `writeSchemaVersion` + adapter 探测 + 只读降级缓解。
- [删除联动误判导致孤儿] → 继续使用现有风险分级与挑战确认，并在写后执行残留校验。
- [helper 与 app 版本错配] → 沿用现有版本一致性前置校验，写操作复用同一门禁。
- [操作成功但 UI 未同步] → 写成功后强制回刷 dump，并以服务端结果覆盖本地临时状态。

## 迁移计划

1. 引入写协议骨架与 capability 字段，默认 `supportsWriteOperations=false`，不改变当前用户可见行为。  
2. 实现 helper 写适配器（按系统版本探测 schema），在支持环境打开写能力。  
3. 将开关与删除切到真实 helper 写链路，保留原确认与结果视图。  
4. 增加回归测试（成功、权限不足、schema 不支持、写后残留）并灰度验证。  
5. 回滚策略：若线上出现不兼容，立即将 capability 强制降为只读，UI 自动禁用写操作。
