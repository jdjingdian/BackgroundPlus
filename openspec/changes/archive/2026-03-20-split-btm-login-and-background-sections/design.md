## 上下文

当前应用已经具备 BTM 条目解析与列表展示能力，但入口与展示仍以“单一后台模块列表”为中心。`sfltool dumpbtm` 输出中同时包含用户登录启动相关条目与后台允许常驻执行条目，语义不同但目前混在同一列表，导致用户在排查某一类问题时成本较高。

本次变更属于现有能力的行为级调整：
- 入口信息架构从单入口变为双入口（登录时打开 / 允许在后台）。
- 同源数据需要增加分类投影层，避免复制解析逻辑。
- 列表行交互（开关、详情）和 helper 依赖语义必须保持稳定，避免回归。

## 目标 / 非目标

**目标：**
- 将 BTM 数据在 UI 层拆分为两个语义明确的侧栏分组并分别展示。
- 定义稳定的分类规则，使同一原始条目可被确定地归入对应分组。
- 保持现有列表行能力、详情跳转与错误处理语义一致。
- 对“某一分组无数据”提供独立空状态，不影响另一分组可用性。

**非目标：**
- 不改变底层 dumpbtm 读取方式与 helper 安装机制。
- 不在本次变更引入新的写入能力（删除/恢复流程不变）。
- 不重做整套信息架构样式，仅调整与分组相关的入口和数据投影。

## 决策

### 决策 1：引入分类投影层，而不是在视图中临时过滤
- 方案 A（采纳）：在解析结果到视图模型之间新增分类投影，将条目映射到 `loginItems` 与 `backgroundItems`。
- 方案 B（放弃）：在每个页面临时按字段过滤。
- 取舍理由：方案 A 可以集中维护分类规则，减少重复代码并提升可测试性；方案 B 容易造成多处规则漂移。

### 决策 2：侧栏显式提供两个固定入口
- 方案 A（采纳）：侧栏新增两个固定功能项“登录时打开”“允许在后台”，分别绑定对应数据集。
- 方案 B（放弃）：保留单入口，在主区使用 Tab 切换。
- 取舍理由：侧栏双入口与系统设置认知一致，且在返回导航时上下文更清晰。

### 决策 3：保持条目行交互契约不变
- 方案 A（采纳）：分类后沿用既有行组件和交互（开关、箭头详情、图标、状态）。
- 方案 B（放弃）：按分组重写两套行交互。
- 取舍理由：方案 A 可降低回归风险，避免重复实现并保持可访问性语义一致。

### 决策 4：分类失败时采用“可用优先”降级
- 方案 A（采纳）：无法可靠分类时将条目标记为 `unknownCategory`，默认落入“允许在后台”并提示分类不完整。
- 方案 B（放弃）：遇到不可分类条目直接丢弃。
- 取舍理由：条目不可丢失，优先保证可见性和可操作性。

## 风险 / 权衡

- [分类规则与系统真实语义存在偏差] → 通过 fixture 覆盖典型类型（app/developer/legacy daemon）并提供规则可扩展点。
- [双入口导致状态管理复杂度上升] → 统一使用一个源数据 + 分类投影，避免维护两份状态。
- [已有 UI 自动化依赖旧入口文案] → 同步更新测试选择器与本地化键，保留兼容映射窗口。
- [某分组空数据被误判为解析失败] → 区分“空分组状态”与“解析失败状态”，分别使用不同提示文案。

## 迁移计划

1. 增加分类投影类型与映射逻辑，并补齐单元测试。
2. 侧栏引入双入口，保持原导航状态机与路由兼容。
3. 列表容器按当前入口读取对应数据集，接入分组空状态。
4. 回归验证 helper 未安装、解析不完整、行开关和详情跳转。

回滚策略：保留单入口渲染分支；若分类逻辑出现严重误判，可临时切回“单列表展示”而不影响底层解析与数据读取。

## 待确认问题

- legacy daemon 在特定系统版本上的归类规则是否需要按 `Disposition` 进一步细分。
- 侧栏文案是否统一采用系统术语（“登录项”）还是保持当前产品术语（“登录时打开”）。
