## 上下文

当前设置窗口中 Helper 安装界面主要关注功能可用，但缺少标准窗口关闭入口，导致用户只能依赖 `Esc` 键退出。该行为在 macOS 桌面应用中不符合常见交互预期。与此同时，“通用”与“关于”页面在视觉层级、对齐节奏与信息分组上存在不一致，影响感知质量。

该变更涉及同一设置窗口下的多个视图：导航切换区、Helper 状态区、操作区和关于信息区。目标是在不引入新后端接口的前提下，通过 UI 结构调整和组件样式统一提升可用性与一致性。

## 目标 / 非目标

**目标：**
- 在 Helper 安装页面提供可见、可点击的关闭按钮，并与现有 `Esc` 行为兼容。
- 重构“通用”与“关于”页的版式与视觉规范，使其与参考图一致（顶部导航、内容分区、信息卡与底部操作区）。
- 统一 Helper 状态展示与安装/卸载操作反馈，明确已安装与未安装状态。

**非目标：**
- 不改动 Helper 安装底层机制（SMJobBless 调用与权限流程）。
- 不引入新的网络请求或远程配置。
- 不变更非设置窗口的其他业务页面视觉风格。

## 决策

### 决策 1：将关闭能力显式化为页面级主动作之一
- 方案：在 Helper 安装页右下角提供明显的关闭按钮（与卸载按钮分离），并保留 `Esc` 作为辅助快捷键。
- 原因：提高可发现性，降低用户学习成本，符合桌面窗口常见交互。
- 备选方案：仅保留快捷键提示，不增加按钮。
- 放弃原因：提示文本不如直接可点击控件直观，无法满足“可见关闭入口”诉求。

### 决策 2：抽象设置窗口通用布局容器
- 方案：提炼统一的设置页容器（顶部标签区 + 主内容区 + 底部操作区），让“通用/关于”共享间距、分割线和按钮规范。
- 原因：减少重复样式，保证后续页面扩展一致性。
- 备选方案：分别在两个页面内局部修补。
- 放弃原因：易造成风格再次分裂，维护成本更高。

### 决策 3：Helper 状态与操作采用“状态优先”信息层
- 方案：将状态文本、状态图标、说明文案与动作按钮放在同一信息块，按钮按状态切换可用性与文案。
- 原因：用户先理解当前状态，再执行动作，认知路径更自然。
- 备选方案：状态与按钮分散在不同区域。
- 放弃原因：信息关联弱，容易误操作。

## 风险 / 权衡

- [风险] 新布局可能影响不同窗口尺寸下的自适应表现。→ 缓解措施：采用最小宽度约束与内容优先的垂直流式布局，并在常见分辨率下验证。
- [风险] 新增关闭按钮后与卸载按钮位置关系可能引发误触。→ 缓解措施：通过颜色和间距区分危险操作与普通操作，关闭按钮使用中性样式。
- [权衡] 追求与参考图一致可能需要较多样式重构。→ 缓解措施：先抽象通用布局组件，减少一次性硬编码。

## Migration Plan

1. 先引入通用设置页容器与共享样式变量，不改变现有功能逻辑。
2. 将“通用”页迁移到新容器，并接入 Helper 状态信息块与关闭按钮。
3. 将“关于”页迁移到新容器，统一标题区、应用信息区与底部操作区。
4. 进行回归验证（安装/卸载流程、`Esc` 退出、按钮状态、窗口缩放）。
5. 若出现重大 UI 回归，可回滚到旧容器实现并保留状态逻辑改动。
