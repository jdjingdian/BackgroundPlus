## 上下文

当前 helper 读取 BTM 存储存在两类问题：
1. 访问前置条件不透明：helper 虽以 root 运行，仍可能因系统访问控制返回 `EPERM`，导致误判为普通文件权限问题。
2. 直读解析语义偏差：`CFKeyedArchiverUID` 解码兼容、`disposition/type` 映射及分类规则与 `sfltool dumpbtm` 存在偏差，表现为启用状态和“登录时打开/允许在后台”分组不一致。

本次变更目标是把已验证结论和已落地修复沉淀到规范层，并补齐可观测性约束，降低后续回归成本。

## 目标 / 非目标

**目标：**
- 明确记录 BTM 直读的环境前置条件：开启 Full Disk Access（FDA）后 helper 才能稳定访问 BTM 文件。
- 固化 helper 的诊断与对比机制，确保可快速定位“访问层”与“解析层”问题。
- 固化分类规则修正，减少将后台 helper 子项误归类为“登录时打开”的情况。

**非目标：**
- 不引入新的系统授权流程或私有 entitlement。
- 不重写完整 BTM 领域模型，仅修正当前已验证的偏差点。
- 不改变 UI 信息架构与导航布局。

## 决策

1. 决策：将 FDA 作为 helper 直读 BTM 的显式前置条件写入规范。
- 原因：实测表明 root 身份不足以保证读取权限，且该结论直接影响写能力协商与用户引导。
- 备选方案：继续仅依赖 root 身份。已否决，因在真实环境不成立。

2. 决策：保留 `btm_file` 与 `sfltool` 双源对比日志能力。
- 原因：两者统计差异能快速定位偏差来源（访问、解析或分类），显著降低排障成本。
- 备选方案：只保留单源日志。已否决，定位效率不足。

3. 决策：分类规则增加“LoginItems helper 子项优先后台化”判定。
- 规则：`url` 命中 `Contents/Library/LoginItems/` 或 `identifier` 前缀为 `4.` 时优先归类后台项。
- 原因：与系统“登录项与扩展”展示语义更一致，已由 `4.com.shreklaunch.mouseboostpro` 案例验证。
- 备选方案：仅按 `type=app` 判定登录项。已否决，误判率高。

4. 决策：当 `parentIdentifier` 缺失时回退 `container`。
- 原因：BTM 数据中大量父子关系通过 `container` 表达，忽略会损失分类和关联分析信息。
- 备选方案：仅使用 `parentIdentifier`。已否决，信息不完整。

## 风险 / 权衡

- [风险] 用户未开启 FDA 时，直读能力退化或不可用。 → 缓解：能力探测中明确只读/降级状态，并保留 fallback 与可读提示。
- [风险] 基于路径/前缀的分类规则仍可能覆盖不全。 → 缓解：保留“完整 Dump 数据”分组及诊断日志，持续收集边缘样本迭代规则。
- [风险] 解析映射随系统版本变化漂移。 → 缓解：保留双源统计对比，出现大幅差异时自动暴露异常信号。

## Migration Plan

- 本次为增量修订，无需数据迁移。
- 发布后按以下顺序验证：
  1. 未开启 FDA：确认 helper 直读受限且状态提示明确。
  2. 开启 FDA：确认 `btm_file` 读取成功。
  3. 对比日志：确认 `entries` 与 identifier 集合一致，enabled/disabled 差异收敛。
  4. 分类回归：确认 `4.*` + `Contents/Library/LoginItems/*.app` 子项落入“允许在后台”。
- 回滚策略：若出现分类误伤，回滚新增分类优先规则，保留对比日志与 FDA 结论文档化成果。
