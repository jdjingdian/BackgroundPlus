## 修改需求

### 需求:BTM 条目可视化
系统 SHALL 提供以“侧栏功能分组 + 主区域列表”为结构的 BTM 条目展示。侧栏必须至少包含“登录时打开”“允许在后台”“完整 Dump 数据”三个入口；主区域列表必须仅展示当前入口对应的条目集合，并包含每行的 icon、主标题、副标题、开关状态与行级 `→` 操作入口。当 helper 已安装时，数据源必须来自真实系统 dump 结果而非固定占位数据。行内开关必须优先反映系统真实状态；当真实写能力不可用时，系统必须禁用写入并明确提示只读状态。系统 MUST 对 helper 直读结果与 `sfltool` 结果提供可比对的诊断统计，且在分类上避免将 LoginItems helper 子项误归类到“登录时打开”。

#### 场景:进入登录时打开分组
- **当** 用户在侧栏选择“登录时打开”
- **那么** 系统必须只展示满足“已确认分类为登录时打开且开关为开启状态”的条目

#### 场景:进入允许在后台分组
- **当** 用户在侧栏选择“允许在后台”
- **那么** 系统必须只展示满足“已确认分类为允许在后台且开关为开启状态”的条目

#### 场景:LoginItems helper 子项归类为后台项
- **当** 条目 URL 命中 `Contents/Library/LoginItems/` 或 identifier 具有 `4.` 前缀
- **那么** 系统必须优先将该条目标记为“允许在后台”，不得归入“登录时打开”

#### 场景:分组空状态独立展示
- **当** 当前分组没有可展示条目且用户进入该分组
- **那么** 系统必须展示该分组空状态提示，且不得影响另一个分组可访问性

#### 场景:分类不完整时可用优先
- **当** dumpbtm 输出存在缺字段或格式漂移导致部分条目分类不确定
- **那么** 系统必须提示“分类不完整”，并将不确定条目仅保留在“完整 Dump 数据”分组中展示

#### 场景:helper 未安装
- **当** 用户打开任一 BTM 分组且 helper 尚未安装
- **那么** 系统必须展示“需先安装 helper”的状态提示和前往设置的引导

#### 场景:行内开关与跳转互不干扰
- **当** 用户点击开关或点击 `→` 按钮
- **那么** 系统必须仅触发对应操作且不得误触发另一个控件行为

#### 场景:写能力不可用时阻断开关写入
- **当** helper 已安装但写能力协商结果为不可写
- **那么** 系统必须阻断开关写入并展示只读提示

#### 场景:开关写入成功后与系统状态一致
- **当** 用户切换某条目开关并收到写入成功回执
- **那么** 系统必须刷新 dump 数据并展示刷新后的真实状态

#### 场景:BTM 直读与 sfltool 统计可对齐
- **当** helper 同时输出 `btm_file` 与 `sfltool` 的对比诊断日志
- **那么** 两者条目总数与 identifier 集合必须一致，enabled/disabled 差异必须可解释且可收敛
