## 上下文

`BackgroundPlus` 目前已经有 BTM 领域模型、解析器、风险确认与备份策略，但应用端缺少可用的特权 helper 链路。现状导致 UI 只能停留在骨架或模拟数据层，无法稳定读取真实系统 BTM 数据，也无法为后续写操作提供可信执行通道。

本次变更跨越 Xcode target 配置、权限签名、XPC 通信、设置引导和主界面数据流，属于多模块架构变更。约束包括：
- helper 必须以 `SMJobBless` 方式安装，Bundle ID 固定为 `cn.magicdian.BackgroundPlus.helper`。
- App 端不能直接执行高权限 BTM 修改，应通过 helper 暴露的受控接口完成。
- UI 需要向用户透明展示 helper 安装状态和失败原因，避免“静默失败”。

## 目标 / 非目标

**目标：**
- 新增 helper target 并完成最小可用通信链路（安装、状态检查、调用）。
- 在设置界面提供 helper 安装引导与状态反馈。
- 主界面展示通过 dump 解析得到的真实 BTM 数据列表。
- 为未来写操作保留统一的 helper 服务边界，避免 App 端权限扩散。

**非目标：**
- 本次不完整实现所有 BTM 写操作矩阵（仅确保读取与基础调用链路可用）。
- 本次不引入远程同步、云备份或自动恢复流程。
- 本次不改变既有风险分级模型的语义定义，仅调整其底层执行通道。

## 决策

### 决策 1：采用双 target 架构（App + blessed helper）
- 选择：新增 `cn.magicdian.BackgroundPlus.helper` 独立 target，负责特权任务。
- 原因：权限边界清晰，符合 macOS 特权操作实践，后续写入能力可在 helper 侧集中审计。
- 备选方案：
  - 方案 A：App 内直接调用命令行工具。未采用，因权限不足且安全边界弱。
  - 方案 B：仅依赖非特权读取路径。未采用，无法支撑后续受控写操作。

### 决策 2：设置页作为 helper 生命周期入口
- 选择：将“检测/安装/重试/错误提示”放入单独设置页。
- 原因：降低首次使用门槛，避免在主功能页直接弹出复杂权限流程。
- 备选方案：首次进入列表页自动触发安装。未采用，交互突兀且失败恢复路径不清晰。

### 决策 3：UI 读数据走统一 service 层，优先 helper 通道
- 选择：ViewModel 只依赖 `BTMService` 抽象；服务层优先调用 helper 获取 dump，再解析为领域模型。
- 原因：保持 UI 与权限细节解耦，便于测试注入与降级处理。
- 备选方案：UI 直接执行 shell 并解析。未采用，耦合高且难以演进到特权写操作。

### 决策 4：安装与数据展示状态化建模
- 选择：引入 helper 安装状态机（未安装/安装中/已安装/失败）与数据加载状态（idle/loading/loaded/error）。
- 原因：便于界面明确反馈，减少异步流程下的状态错乱。
- 备选方案：只使用布尔标记。未采用，无法表达失败恢复和中间态。

## 风险 / 权衡

- [SMJobBless 集成复杂，签名或 entitlement 配置错误导致安装失败] → 提供设置页诊断信息与重试入口，文档化证书/签名检查步骤。
- [helper 与 App 协议漂移导致调用失败] → 抽象稳定协议接口并添加协议版本字段，测试覆盖请求/响应解析。
- [特权接口暴露过宽带来安全风险] → helper 侧仅开放白名单方法，参数严格校验并记录审计日志。
- [真实 dump 数据体量大导致首屏卡顿] → 异步加载与分步解析，UI 展示加载态并支持手动刷新。

## Migration Plan

1. 新增 helper target、Info.plist、launchd 配置和签名设置。
2. 在 App 端接入 helper 安装管理器与 XPC 客户端。
3. 新增设置页入口，完成安装引导与状态反馈。
4. 列表页切换为真实数据源（helper dump -> parser -> view model）。
5. 执行本地签名构建与基础 UI/集成测试；失败时保留非破坏性回退开关。

回滚策略：若 helper 集成在发布前不稳定，可在构建配置中关闭 helper 通道并回退到只读/占位模式，不移除既有数据模型与 UI 结构。
