## 上下文

当前 helper 管理流程以“是否安装成功”为主，缺少“版本/接口是否兼容”的前置判断。应用升级后，如果系统仍保留旧版 helper，主程序可能在调用新增接口时阻塞或卡住。该问题涉及主程序启动路径、helper 握手逻辑和安装引导 UI，属于跨模块行为一致性问题。

## 目标 / 非目标

**目标：**
- 在调用任意高风险 helper 能力前，完成版本一致性与能力读取校验。
- 建立一个可长期保留的 helper 能力读取接口，作为 app 与 helper 兼容性协商入口。
- 当检测到版本不匹配或能力读取失败时，统一进入“强提示并要求重装 helper”的受控流程。
- 避免旧 helper 与新 app 组合导致的主进程卡住。

**非目标：**
- 不在本次变更中重构全部 helper API。
- 不引入自动静默重装；仅提供明确阻断与用户触发的重装路径。
- 不改变 BTM 业务语义，仅增强调用前健康检查与错误处置。

## 决策

1. 引入启动前握手检查（版本 + 能力）
- 决策：在 app 启动后首次使用 helper 前执行握手，读取 helper 版本与能力描述，和 app 期望版本进行比对。
- 原因：把风险前移到调用前，避免在实际特权调用中暴露卡死问题。
- 备选：
  - 仅在接口失败后回退提示：太晚，用户已进入异常路径。
  - 每次调用都检查：开销更高且重复，可通过缓存握手结果优化。

2. 固化 helper 能力读取接口为稳定契约
- 决策：定义并保留一个最小能力读取接口（至少包含 helper 版本与接口版本信息），后续版本不得移除。
- 原因：没有稳定握手接口就无法做兼容判断，只能盲调高风险接口。
- 备选：
  - 通过文件或 plist 读取版本：信息可能与运行时能力脱节。
  - 依赖 install metadata：无法覆盖手工替换或残留旧 helper 场景。

3. 版本不匹配时强阻断 + 强提示重装
- 决策：检测到不匹配或握手失败时，阻断后续特权调用，并在 UI 给出高优先级提示（说明风险、重装入口、完成后重试）。
- 原因：继续执行会造成不可预测行为和卡死，强阻断可保证系统一致性。
- 备选：
  - 降级继续运行：风险不可控。
  - 静默忽略：用户无法理解失败原因，支持成本高。

## 风险 / 权衡

- 误判兼容性导致过度阻断 → 明确版本匹配规则（精确匹配或兼容窗口）并记录诊断日志。
- 强提示增加操作成本 → 在提示中提供清晰的一键重装路径和失败重试指引。
- 稳定接口长期保留带来维护负担 → 将接口保持最小化，只承载版本与能力元信息。

## 迁移计划

1. helper 先提供稳定能力读取接口与版本返回。
2. app 接入握手检查，并将高风险调用前置到“已通过兼容校验”状态。
3. UI 增加版本不匹配强提示与重装入口。
4. 灰度验证：覆盖“旧 helper + 新 app”“能力读取失败”“重装后恢复”三类场景。
5. 回滚策略：若新检查引发异常，可临时回退到旧版本 app/helper 组合，但不放开不兼容调用。

## 开放问题

- 版本匹配策略采用严格相等，还是允许小版本兼容窗口。
- 强提示交互是否要求阻断除重装外的全部设置项操作。
