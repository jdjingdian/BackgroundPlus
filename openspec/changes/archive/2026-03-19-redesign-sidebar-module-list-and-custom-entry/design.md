## 上下文

当前 BTM 管理页面把功能导航和数据列表混在侧边栏中，导致信息结构不清晰：用户无法快速判断“我在切换功能”还是“我在浏览条目”。
本次变更需要将信息架构调整为接近 macOS 系统设置的模式：侧边栏用于功能切换，主区域承载后台模块列表与操作。与此同时，用户提出要在列表每一行提供进入“我的定制功能”的快捷入口，并复用现有 app 详情能力。

约束：
- 不改变 BTM 数据来源、解析和开关核心逻辑。
- 需保持已有删除、风险确认、备份能力可达。
- UI 风格优先采用系统原生控件与图标渲染方式，避免自定义重绘导致视觉不一致。

## 目标 / 非目标

**目标：**
- 将主界面重构为“侧栏功能导航 + 主区模块列表”的双栏信息架构。
- 在模块列表中展示 app 图标、主标题、副标题、开关状态，并保证开关交互不退化。
- 在每一行开关右侧提供 `→` 进入按钮，进入“我的定制功能”详情页。
- 详情页沿用当前应用已有 app 信息展示结构，避免重复设计。

**非目标：**
- 不重写 BTM 解析器、删除引擎或 helper 安装流程。
- 不引入新的后台权限模型。
- 不对“我的定制功能”详情内容做语义扩展，仅调整入口路径与导航方式。

## 决策

1. 采用导航状态与内容状态解耦
- 决策：侧栏状态仅表示当前功能（如“后台模块”“设置”“关于”），不再绑定单条 BTM 项。
- 原因：避免侧栏项目数量随数据量膨胀，保持可预测导航体验。
- 备选方案：继续用侧栏承载条目并在右侧展示详情。否决原因：与系统设置的心智模型冲突，且难以扩展更多功能页。

2. 主区使用统一列表行组件承载模块信息
- 决策：将 icon、标题、副标题、右侧操作（开关 + `→`）收敛到同一行组件。
- 原因：减少多组件拼接导致的间距和点击区域不一致。
- 备选方案：开关单列、跳转按钮放次级菜单。否决原因：多一步操作，降低发现性。

3. 图标优先使用系统原生来源
- 决策：通过系统 API 获取 app 图标（可回退到系统默认占位 icon），并遵循系统尺寸与圆角风格。
- 原因：与 macOS 原生视觉一致，避免自定义素材维护成本。
- 备选方案：自绘 SVG 或统一自定义图标包。否决原因：与原生风格偏差明显。

4. 行内 `→` 跳转采用显式路由参数
- 决策：点击行内 `→` 后携带 entry/app 标识进入“我的定制功能”页，并复用 `BackgroundItemDetailView`（MVVM），在返回时保持列表滚动和选中上下文。
- 原因：保证从列表到详情再返回的连续性。
- 备选方案：弹窗展示详情。否决原因：不利于承载完整 app 信息，且与现有主页面结构冲突。

## 风险 / 权衡

- [风险] 列表行内同时存在开关与跳转按钮，误触概率上升
  → 缓解措施：扩大控件间距，分离点击热区，键盘焦点顺序明确。

- [风险] 图标读取失败造成视觉噪声
  → 缓解措施：提供稳定的默认占位 icon，并记录失败原因用于调试。

- [风险] 导航重构影响已有详情面板状态同步
  → 缓解措施：引入单向状态源（当前功能、当前条目、当前详情目标）并补充回归测试。

- [权衡] 更接近系统设置布局会牺牲部分“一屏展示全部条目详情”的密度
  → 缓解措施：在主区提供清晰摘要信息，详情通过 `→` 快速进入。

## Migration Plan

1. 第 1 步：重构主容器，把侧栏数据源替换为固定功能项；保留旧列表组件但移动到主区。
2. 第 2 步：实现新列表行组件（icon/标题/副标题/开关/`→`），接入现有开关动作。
3. 第 3 步：实现 `→` 跳转到“我的定制功能”路由，并复用现有 app 详情渲染。
4. 第 4 步：补充 UI 回归检查（空图标、helper 未安装、解析不完整、长文本截断）。
5. 回滚策略：保留旧容器开关，在发现严重回归时切回旧布局组件树，不影响底层 BTM 操作链路。

## Open Questions

- `→` 按钮是否在所有条目都显示，还是仅对可识别 app 的条目显示？
- 当条目缺少可用 app 标识时，详情页是展示降级信息还是禁止跳转？
- 是否需要在列表顶部提供筛选/搜索，以应对条目数量较大场景？
