## 上下文

当前 BTM 管理页在“解析不完整”和“分类不完整”同时出现时，通常会堆叠多个提示块，造成页面顶部信息密度过高。用户反馈希望参考现有 `helperBanner` / `StatusBanner` 的实现方式，把相关提示收敛为单行 banner，以获得更稳定的视觉节奏。

该变更只涉及 BTM 列表顶部的状态提示聚合与展示方式，不改变条目解析主流程、列表分组规则、详情页跳转和开关写入逻辑。

## 目标 / 非目标

**目标：**
- 将“解析不完整”和“分类不完整”统一为一个顶部 banner 区域，且任意时刻最多展示一条。
- 在两种状态并存时输出组合文案，避免重复占用纵向空间。
- 复用与 `StatusBanner` 一致的结构（图标 + 文案 + 可选动作区），保证视觉一致性和可维护性。

**非目标：**
- 不调整 BTM 原始解析算法与分类判定标准。
- 不修改 helper 安装状态 banner 的优先级逻辑。
- 不引入新的页面级通知体系或 toast 组件。

## 决策

1. 在 ViewModel 引入“告警聚合状态”作为单一数据源
- 决策：新增聚合枚举（例如 `none / parseOnly / classifyOnly / parseAndClassify`），由已有解析标志推导，视图层只消费该聚合状态。
- 原因：避免视图里散落多处 `if` 判断，减少状态组合遗漏。
- 备选方案：在 SwiftUI 视图中直接拼接多个条件分支；实现快但可读性差，后续扩展文案风险高。

2. 采用单一 Banner 组件渲染（复用 StatusBanner 模式）
- 决策：沿用 `StatusBanner` 风格参数化能力，BTM 侧只传入 `warning` 样式和聚合后的本地化 key。
- 原因：与现有顶部提示视觉统一，且无需重复实现布局与分隔线。
- 备选方案：为 BTM 单独实现一个新提示组件；会产生重复 UI 代码，维护成本更高。

3. 文案策略采用“组合优先，单态兜底”
- 决策：优先支持三类 key：`parse_only`、`classify_only`、`parse_and_classify`，按聚合状态一一映射。
- 原因：避免运行时字符串拼接带来的本地化和语义问题。
- 备选方案：动态拼接两段文案；在多语言下顺序和连词不稳定。

## 风险 / 权衡

- [风险] 聚合状态映射遗漏导致本应提示的异常不显示。
  -> 缓解：为四种聚合状态补充单元测试/快照断言，并在默认分支回退到最保守提示。

- [风险] 组合文案过长导致窄窗口下可读性下降。
  -> 缓解：保持单行优先并允许文本截断后悬停查看完整内容，必要时优化 key 长度。

- [权衡] 合并提示会降低“每类异常独立显式”的粒度。
  -> 缓解：在组合文案中保留两类风险关键词，确保用户仍能识别问题来源。

## Migration Plan

1. 在 BTM 状态模型中新增聚合告警状态并接入现有解析结果。
2. 调整 BTM 顶部提示渲染逻辑：从多条件多提示改为单条件单 banner。
3. 新增/更新本地化 key，覆盖三种文案组合。
4. 回归验证：
   - 仅解析不完整
   - 仅分类不完整
   - 两者同时存在
   - 两者均不存在
5. 回滚策略：若出现告警丢失，可暂时回退到原先独立提示逻辑。
