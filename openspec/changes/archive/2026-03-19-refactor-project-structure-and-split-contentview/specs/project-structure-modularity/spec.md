## ADDED Requirements

### Requirement: UI 分层与目录边界
系统 MUST 将应用界面代码按“壳层入口、页面容器、复用组件、弹层视图”进行明确分层，并为每类代码提供稳定目录边界，禁止将多类职责长期集中在单一入口视图文件中。

#### Scenario: 新增页面功能时的落点约束
- **WHEN** 开发者新增或调整页面功能
- **THEN** 新代码必须落在对应职责目录中，且不得将页面容器逻辑直接堆叠到壳层入口文件

### Requirement: ContentView 职责收敛
系统 MUST 将 `ContentView` 作为应用壳层入口，仅承载顶层布局与依赖装配；业务流程编排和具体区域交互必须由独立容器视图承接。

#### Scenario: 入口视图职责验证
- **WHEN** 系统完成本次重构
- **THEN** `ContentView` 不再直接包含多个业务区域的详细交互实现，且业务交互由独立容器模块提供

### Requirement: 行为一致性保障
系统 MUST 在目录重构与组件拆分后保持现有用户可见行为一致，包括路由管理、分组编辑、设置入口与 helper 状态展示结果，不得因结构调整引入功能缺失。

#### Scenario: 重构后关键路径回归
- **GIVEN** 用户执行现有核心操作流程
- **WHEN** 系统运行在重构后的代码结构上
- **THEN** 用户可见结果与重构前保持一致
