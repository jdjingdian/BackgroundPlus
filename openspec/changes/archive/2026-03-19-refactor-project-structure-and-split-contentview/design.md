## 上下文

当前 `StaticRouter` 已具备 `Model`、`Services`、`View` 等目录，但 UI 编排仍存在向 `ContentView` 集中的趋势，导致单文件职责过重：既包含页面结构，又承载状态切换与部分流程编排。参考同类项目实践，应将“页面容器、业务状态、可复用子视图”拆分为可组合单元，并通过清晰的目录约定降低查找与维护成本。

约束如下：
- 不改变现有核心行为（路由列表管理、分组编辑、设置入口、Helper 状态展示等）。
- 重构期间保持编译通过，避免一次性大迁移造成长时间不可用。
- 现有 `Model` 与 `Services` 的公共接口尽量稳定，优先调整调用组织，而非同时重写底层逻辑。

## 目标 / 非目标

**目标：**
- 建立一致的 UI 分层：容器视图负责状态编排，展示视图负责渲染与交互。
- 将 `ContentView` 拆分为多个职责明确的子模块，控制单文件复杂度。
- 明确目录边界，使新页面和新组件有固定落点，减少“无处可放”的增量代码。
- 在不改变用户可见行为前提下，提升可测试性与可维护性。

**非目标：**
- 不引入新的产品功能或交互流程。
- 不修改核心网络/系统写入逻辑（例如特权操作实现细节）。
- 不在本次变更中全面替换状态管理范式（如引入全新架构框架）。

## 决策

### 决策 1：以“按职责分层 + 按功能聚合”的混合目录策略重组 UI 代码
- 方案：在 `View` 内明确 `Screens`（页面容器）、`Components`（可复用展示单元）、`Sheets`/`Panels`（弹层）等子层级；跨页面可复用控件保留在通用组件目录。
- 原因：纯按技术类型分层（全部按 ViewModel/View）会让业务上下文分散；纯按功能分层在共享组件较多时会重复。混合策略可兼顾可发现性与复用性。
- 备选方案：
  - 全量按功能模块拆目录：边界清晰但初期迁移成本高。
  - 维持现状仅做文件切分：短期成本低，但结构问题会继续累积。

### 决策 2：将 `ContentView` 降级为应用壳层入口（Shell），其内部只做路由与高层布局
- 方案：`ContentView` 只保留窗口主框架与顶层依赖注入，具体业务区域由独立容器（如侧栏容器、列表容器、设置容器）承接。
- 原因：将高层入口与具体业务拆开后，功能演进时可以局部替换，降低冲突。
- 备选方案：保留大型 `ContentView` 并通过 `extension` 切分。该方案文件仍强耦合，不利于后续按功能演进。

### 决策 3：优先“稳定接口、迁移调用点”，而非同步改造模型和服务
- 方案：先迁移 UI 组织和依赖方向；如需调整 `Services`/`Model`，仅做最小接口封装。
- 原因：重构风险主要来自 UI 回归与状态流变化。控制变量可缩短验证链路。
- 备选方案：UI 与服务层同时重构。虽然理论上一步到位，但风险与排查成本显著增加。

## 风险 / 权衡

- [拆分后状态传递链路变长，导致状态不同步] → 在容器层定义单一状态源，展示层通过参数下发，避免多点持有。
- [文件移动引发引用或预览失效] → 采用小步迁移，每一步执行构建验证并修复编译错误后再继续。
- [重构过程中行为偏差难以及时发现] → 针对关键路径建立手工回归清单（启动、增删改路由、切换设置、Helper 状态展示）。
- [目录规范后续被破坏] → 在任务中加入“目录约定文档与命名规则”并在 code review 中强制检查。

## 迁移计划

1. 梳理 `ContentView` 现有职责并标注可独立拆分的 UI 片段与状态入口。
2. 先提取纯展示子视图（无业务副作用），保持行为不变。
3. 再提取容器视图，迁移状态编排与事件转发。
4. 调整文件目录与命名，将新旧视图映射到目标结构。
5. 执行构建与关键路径回归，修复迁移中问题。
6. 若出现不可控回归，按提交粒度回滚到上一个稳定阶段。

## 开放问题

- 是否需要在本次重构中引入统一的 UI 依赖注入入口（例如环境对象聚合），还是保持当前注入方式并仅做目录与职责整理。
- `Shared` 与 `StaticRouter/View/Components` 的边界需进一步统一：哪些组件应提升为跨 target 共享。
