## 为什么

当前项目目录职责边界不清晰，且 `ContentView` 承担了过多 UI 与编排逻辑，导致代码可读性下降、功能迭代成本升高、测试与复用困难。现在进行结构化拆分可以在不改变核心功能的前提下，降低后续开发风险并提升协作效率。

## 变更内容

- 重构 `StaticRouter` 的目录组织，将 UI、业务编排、模型与服务按职责分层，形成稳定且可扩展的工程结构。
- 将 `ContentView` 中过于集中的视图与交互逻辑拆分为多个可复用子视图与容器层，保留现有用户可见行为。
- 对拆分后的模块建立清晰依赖方向，避免视图层直接耦合底层实现细节。
- 补充必要的迁移与验证步骤，确保重构后构建、运行与核心功能一致。

## 功能 (Capabilities)

### 新增功能
- `project-structure-modularity`: 定义项目目录分层与 `ContentView` 组件化拆分的规范化要求，确保后续功能以模块化方式演进。

### 修改功能
- `privileged-helper-management`: 调整该能力在 UI 层的组织与接入方式，要求在重构后保持原有行为与交互结果一致。
- `btm-entry-management`: 调整该能力相关界面在拆分后的模块归属，要求不改变用户可见行为与数据流结果。

## 影响

- 受影响代码：`StaticRouter` 下 `View`、`Components`、`Model`、`Services` 及应用入口附近编排代码（重点为 `ContentView` 与其调用链）。
- 受影响系统：SwiftUI 视图组织方式、模块依赖关系、部分文件路径与命名。
- 对外 API/数据格式：预期无破坏性变化。
- 风险：重构过程可能引入 UI 状态回归或依赖注入遗漏，需要通过构建与关键路径回归验证控制风险。
