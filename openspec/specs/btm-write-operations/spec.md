# btm-write-operations 规范

## 目的
待定 - 由归档变更 btm-single-delete-toggle-write-path 创建。归档后请更新目的。
## 需求
### 需求:Helper 写能力协商
系统必须在执行任何 BTM 写操作前完成 helper 写能力协商；当 helper 未声明支持写能力时，系统必须阻断写操作并降级为只读。

#### 场景:写能力可用
- **当** app 读取到 helper capabilities 且 `supportsWriteOperations=true`
- **那么** 系统必须允许进入开关与删除写入流程

#### 场景:写能力不可用
- **当** app 读取到 helper capabilities 且 `supportsWriteOperations=false`
- **那么** 系统必须阻断所有 BTM 写操作并提示“当前环境仅支持只读”

### 需求:单条开关写操作
系统必须支持针对单个 BTM 条目执行启用/禁用写操作，并在写入成功后刷新真实 dump 数据。helper 能力协商结果必须与该能力实际实现保持一致，禁止出现“能力宣称可写但 toggle 空实现”的状态。系统必须按条目语义选择开关位：登录时打开条目优先切换 `enabled` 位，后台运行条目优先切换 `allowed` 位。

#### 场景:开关写入成功
- **当** 用户切换某条目开关且 helper 写操作返回成功
- **那么** 系统必须刷新条目列表并展示与真实系统状态一致的开关值

#### 场景:开关写入失败
- **当** 用户切换某条目开关且 helper 返回错误
- **那么** 系统必须回滚本地开关显示并展示错误信息

#### 场景:开关能力未实现或 schema 不支持
- **当** helper 无法提供 toggle 写能力（例如 schema 未达到要求）
- **那么** 系统必须阻断开关写入并返回“开关能力不支持”错误，不得误报为权限问题

#### 场景:登录时打开条目切换
- **当** 用户切换 `app (0x2)` 登录项开关
- **那么** 系统必须切换 `enabled` 位（`0x1`），并在 dump 中体现 `enabled/disabled` 变化

#### 场景:后台服务条目切换
- **当** 用户切换 `daemon (0x10)` 或 `legacy daemon (0x10010)` 背景服务开关
- **那么** 系统必须切换 `allowed` 位（`0x2`），并在 dump 中体现 `allowed/disallowed` 变化（即使同一条目在不同环境下 `enabled` 位可能不同）

### 需求:单点删除真实执行
系统必须支持针对单个目标条目执行删除写操作，并保持写前备份与写后校验语义。

#### 场景:删除写入成功
- **当** 用户完成风险确认并触发删除，且 helper 执行成功
- **那么** 系统必须返回成功结果、展示备份路径并在刷新后不再显示目标条目

#### 场景:删除写入失败
- **当** 用户触发删除且 helper 返回执行失败
- **那么** 系统必须展示失败结果并保留可追溯错误信息
